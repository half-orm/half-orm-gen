# Auto-CRUD via `CRUD_ACCESS`

## Overview

For each halfORM relation module that defines a `CRUD_ACCESS` dict,
`half_orm gen generate` produces a complete set of CRUD route handlers —
no manual coding required. Access control (which roles can call which verb, on
which fields) is declared alongside the model, with distinct input and output
field lists per role.

---

## Where to declare `CRUD_ACCESS`

The dict goes in the **module-level developer space** of a half-orm-dev
generated file — between the second pair of `#>>>` / `#<<<` markers (the first
pair is in the module docstring as an example):

```python
# myproject/blog/post.py  (generated by half-orm-dev)

#>>> PLACE YOUR CODE BELOW THIS LINE. DO NOT REMOVE THIS LINE!

API_EXCLUDED_FIELDS = ["passwd_hash"]   # optional — see below

CRUD_ACCESS = {
    "GET":    {"anonymous": ["id", "title", "published"], "connected": None},
    "POST":   {"connected": {"in": ["title", "content", "author_id"]}},
    "PUT":    {"connected": {"in": ["title", "content", "published"]}},
    "DELETE": {"admin": None},
}

#<<< PLACE YOUR CODE ABOVE THIS LINE. DO NOT REMOVE THIS LINE!
```

Omitting `CRUD_ACCESS` entirely means no auto-CRUD routes are generated for
that relation — deny by default.

---

## Structure

### GET (and DELETE)

For `GET` and `DELETE`, the role value is the **output** field list (or `None`
for all fields):

```python
"GET": {
    "anonymous":    ["id", "name", "email"],   # only these fields returned
    "connected": None,                       # all fields returned
}

"DELETE": {
    "admin": None,   # no in/out concept for DELETE
}
```

### POST and PUT

For `POST` and `PUT`, the role value is a dict with explicit `in` and `out`
keys:

```python
"POST": {
    "connected": {
        "in":  ["name", "email", "passwd"],   # accepted fields in request body
        "out": ["id", "name", "email"],        # fields returned in response
    }
}
```

| Key | Meaning |
|-----|---------|
| `"in"` | Fields accepted in the request body. `None` = all non-excluded fields. |
| `"out"` | Fields returned in the response. `None` = all non-excluded fields. Omit to inherit from `GET` for the same role. |

**`out` inheritance**: if `"out"` is omitted on `POST`/`PUT` for a given role,
the generator looks up `CRUD_ACCESS["GET"][role]` and uses its value. If the
role is absent from `GET`, `None` (all fields) is used.

```python
"GET":  {"connected": None},          # out = all fields
"PUT":  {"connected": {"in": ["name", "email", "passwd"]}},
#                      ↑ out omitted → inherits GET["connected"] = None
```

---

## Blocking fields from the entire API

`API_EXCLUDED_FIELDS` is an optional module-level list of fields that are
**never** exposed through the API — neither in responses nor accepted in request
bodies — regardless of role or verb:

```python
API_EXCLUDED_FIELDS = ["passwd_hash", "internal_token"]
```

This overrides `None` (all fields): even when a role grants unrestricted access,
excluded fields are silently removed from inputs and outputs.

Use this for fields that must never leave the server (password hashes, internal
tokens, audit columns managed by triggers, etc.).

---

## Full example

```python
# myproject/actor/user.py

API_EXCLUDED_FIELDS = ["passwd_hash"]

CRUD_ACCESS = {
    # Unauthenticated users see a limited anonymous profile.
    # Authenticated users see everything except the excluded hash.
    "GET": {
        "anonymous":    ["id", "name", "email"],
        "connected": None,
    },

    # Creating an account: accept passwd, never return it.
    "POST": {
        "anonymous": {
            "in":  ["name", "email", "passwd"],
            "out": ["id", "name", "email"],
        }
    },

    # Updating: connected user can change name, email, passwd.
    # Response inherits connected's GET = None (all non-excluded fields).
    "PUT": {
        "connected": {
            "in": ["name", "email", "passwd"],
        }
    },

    # Only admins can delete.
    "DELETE": {"admin": None},
}
```

## Roles

Three kinds of roles can appear in `CRUD_ACCESS`:

| Role | Meaning |
|------|---------|
| `"anonymous"` | No authentication required. |
| `"connected"` | Any authenticated user. |
| `"<name>"` | A domain role defined in `api/roles/<name>.py`. |

Multiple roles can coexist for the same verb. The **effective fields** for a
request are the union across all active roles:

- If any active role has `None` → all non-excluded fields.
- Otherwise → union of the listed fields, minus `API_EXCLUDED_FIELDS`.

---

## Domain roles (`api/roles/`)

Each domain role is a module in `api/roles/` exporting an async `authorize`
function:

```python
# api/roles/admin.py
async def authorize(path_params: dict, jwt_payload: dict) -> bool:
    return jwt_payload.get("role") == "admin"
```

### Role refinement with `@authorize_and` / `@authorize_or`

`api/roles/core.py` (scaffolded automatically) provides two decorators:

```python
from api.roles.core import authorize_and

# "permanent" must also satisfy "membre"
@authorize_and("membre")
async def authorize(path_params, jwt_payload):
    return jwt_payload.get("status") == "permanent"
```

```python
from api.roles.core import authorize_or

# "editor" OR "admin" grants access
@authorize_or("admin")
async def authorize(path_params, jwt_payload):
    return jwt_payload.get("role") == "editor"
```

## Generated routes

For a relation with a single-column primary key:

| Verb | Route |
|------|-------|
| GET | `GET /vN/schema/table` (list, with optional query-param filters) |
| GET | `GET /vN/schema/table/{id: <type>}` (single row) |
| POST | `POST /vN/schema/table` |
| PUT | `PUT /vN/schema/table/{id: <type>}` |
| DELETE | `DELETE /vN/schema/table/{id: <type>}` |

For relations with a composite or absent primary key, only `GET` (list) is
generated. For views, only `GET` routes are generated regardless of what
`CRUD_ACCESS` declares.

### Access map endpoint

A `GET /vN/ho_access` route is always generated alongside the CRUD routes.
It returns the access control map **filtered to the caller's active roles**:
anonymous users get only `anonymous` entries, authenticated users get `anonymous` +
`connected` + their domain roles. The field lists are the union across all
active roles, with `out` inheritance resolved and `API_EXCLUDED_FIELDS` applied.

```json
GET /v0/ho_access        # anonymous caller

{
    "blog/author": {
        "GET": { "out": ["id", "name", "email"] }
    },
    "blog/post": {
        "GET": { "out": ["id", "title", "published", "author_id"] }
    }
}
```

```json
GET /v0/ho_access        # connected caller

{
    "blog/author": {
        "GET":    { "out": ["id", "name", "email"] },
        "POST":   { "in": ["name", "email"], "out": ["id", "name", "email"] },
        "PUT":    { "in": ["name", "email"], "out": ["id", "name", "email"] }
    },
    "blog/post": {
        "GET":    { "out": ["id", "title", "content", "published", "author_id"] },
        "POST":   { "in": ["title", "content", "author_id"], "out": ["id", "title", "content", "published", "author_id"] },
        "PUT":    { "in": ["title", "content", "published"],  "out": ["id", "title", "content", "published", "author_id"] }
    }
}
```

This endpoint is `anonymous` and is the primary tool for frontends to resolve
action buttons and field visibility per user profile, without multiplying pages.

## Interaction with `@api_*`

If a method decorated with `@api_get`, `@api_post`, etc. already covers a
`(module, verb)` pair, auto-CRUD skips that verb for that relation. The
hand-written route takes precedence with its own guards and logic.

---

## Named row filters (`@ho_api_filter`)

`CRUD_ACCESS["filter"]` (see [Structure](#structure) above) is a *static*
row filter — the same dict on every request. `@ho_api_filter` is for filters
that need to look at the request itself (e.g. "published, or mine"):

```python
from half_orm_gen.tools import ho_api_filter

class Post(ho_baseclasses.BC_BlogPost):

    @ho_api_filter('published_posts')
    def _is_published(self, request):
        user = getattr(request.state, 'user', None)
        visible = self.__class__(published=True)
        if user:
            visible |= self.__class__(author_id=user)   # own drafts too
        return self & visible
```

**Signature**: `(self, request) -> Relation`. `self` already carries the
query's other constraints (pagination, column filters, role filter) —
narrow it with halfORM's predicate algebra (`&`, `|`) rather than replacing
it, so those other constraints survive.

**Wiring**: at startup, `discover_and_register` scans relation modules for
`@ho_api_filter` methods, registers them by `(schema, table, filter_name)`,
and inserts the filter name into `half_orm_meta.api.filter`. A filter only
runs for a request when the caller's most specific matched role lists its
name under `CRUD_ACCESS`/access-map `"filters"` for that verb — configure
this the same way you configure roles, in the admin UI. It runs *after* the
static `role_filter` (from `CRUD_ACCESS["filter"]`) and *before* the query
executes, on both the list (`GET` collection) and single-row (`GET {id}`)
handlers.

See also: [relation-class-guide.md](relation-class-guide.md) for when a
filter has outgrown "this relation's own predicate."

## Hand-written routes (`@tools.api_get`/`api_post`/`api_put`/`api_delete`/`api_patch`)

For a one-off endpoint that doesn't fit the CRUD/verb model — a nonstandard
path, a side effect, a response shape auto-CRUD can't express — decorate a
method directly on the relation class:

```python
from half_orm_gen import tools

class Comment(ho_baseclasses.BC_BlogComment):

    @tools.api_post('/blog/comment/{id: uuid}/flag', guards=['connected'])
    async def flag(self, request: "Request", id: "uuid.UUID") -> dict:
        await self.__class__(id=id).ho_aupdate(flagged=True)
        return {'ok': True}
```

**Signature parity**: each `@tools.api_*` decorator mirrors the signature of
the underlying Litestar decorator it wraps (`litestar.get`/`post`/`put`/
`delete`/`patch`) — so `guards=`, `path` parameters, and any other Litestar
kwarg work exactly as they would on a plain `@get`/`@post`/etc. The method
is called on a **fresh instance** of the relation class (`Comment()`, not a
specific row) — use path/query parameters to build whatever predicate the
method needs, as `flag` does above with `id`.

**Precedence**: if a `@tools.api_*` method already covers a `(module, verb)`
pair, auto-CRUD skips generating that verb for the relation — the
hand-written route wins, with its own guards and return shape.

See also: [relation-class-guide.md](relation-class-guide.md) for when a
custom route has outgrown "this relation" and belongs in
`ho_api/custom/routes.py` instead.

## OpenAPI documentation

Each generated route includes a `description` summarising the access control
for that verb. The format reflects the `in`/`out` distinction:

**GET** (sugar form):
```
**Access**
- anonymous: id, name, email
- connected: all fields
```

**POST** (explicit `in`, `out` inherited from GET and resolved):
```
**Access**
- connected: in=[name, email], out=[id, name, email]
```

**POST** (explicit `in` and `out`):
```
**Access**
- connected: in=[name, email, passwd], out=[id, name, email]
```

**DELETE**:
```
**Access**
- admin: all fields
```

This is visible in the Swagger UI and any OpenAPI-compatible tool.
