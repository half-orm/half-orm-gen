# What goes in a relation class

> `@ho_api_role`: [authorization.md](authorization.md)

## Overview

A generated relation module (e.g. `myproject/blog/post.py`) is regenerated by
`half-orm-dev` on every `hop update`, but the code you write between the
`#>>>`/`#<<<` markers is preserved. That space tends to accumulate: a
`CRUD_ACCESS` dict here, a dynamic role there, a one-off route six months
later. This page draws the line between what belongs on the class and what
doesn't, so that line gets decided once instead of relation by relation.

---

## What belongs on the class

Everything below only ever needs `self` — this relation's own rows and
columns — and, for request-scoped logic, `request`. That's the test: if it
fits that shape, it's a relation-class citizen.

| Construct | Purpose | Docs |
|---|---|---|
| `CRUD_ACCESS` | Declarative per-verb, per-role, per-field access | [crud-access.md](crud-access.md#where-to-declare-crud_access) |
| `API_EXCLUDED_FIELDS` | Fields hidden from the whole API, every verb | [crud-access.md](crud-access.md#blocking-fields-from-the-entire-api) |
| `Fkeys` | halfORM's own FK alias dict (not half-orm-gen specific) — gives foreign keys short, stable names instead of the generated `_reverse_fkey_...` / constraint names | printed in the class docstring by `hop update`; see any generated relation file |
| `@ho_api_role` | Dynamic, row-scoped role resolver (e.g. "is this row's author") | [authorization.md](authorization.md#dynamic-roles-ho_api_role) |
| `@ho_api_filter` | Named row filter applied to a GET query | [crud-access.md](crud-access.md#named-row-filters-ho_api_filter) |
| `@tools.api_get` / `api_post` / `api_put` / `api_delete` / `api_patch` | One-off, hand-written routes bound to this relation | [crud-access.md](crud-access.md#hand-written-routes-toolsapi_) |

## What belongs in `ho_api/custom/` instead

`ho_api/custom/middlewares/authorization.py` and `ho_api/custom/routes.py`
are **never touched by the generator** — create them by hand
(see [../internals/litestar-architecture.md](../internals/litestar-architecture.md#litestar-default)).
That's where cross-cutting logic goes: authentication flows, request
middleware, routes that join or orchestrate across several
relations/silos, or anything that isn't naturally one relation's query or
mutation.

The blog demo's `POST /auth/signup`, `POST /auth/login` and `GET /ho_users`
routes are a good example — none of them is "a query or mutation on one
relation," so they live in `ho_api/custom/routes.py`
(see [authorization.md — Blog demo example](authorization.md#blog-demo-example)),
not on `actor/user.py`.

---

## The dividing line

> If the logic only needs `self` (this relation's own rows) and, for
> request-scoped logic, `request`, it belongs on the relation class as a
> decorator. The moment it needs to join across relations, orchestrate
> multiple silos/models, or isn't naturally expressed as a single relation's
> query or mutation, move it to `ho_api/custom/routes.py` and call the
> relation classes from there instead.

### Smell test, not a hard limit

There's no fixed number of decorators that's "too many." Watch for:

- Several unrelated `@tools.api_*` routes piling up on one relation — each
  one might be legitimate alone, but a growing, unrelated collection is a
  sign they should be split into `custom/routes.py` handlers that call the
  relation classes they need.
- A `@ho_api_filter` or `@ho_api_role` that starts querying *other* tables
  to make its decision — at that point it's no longer "this relation's own
  predicate," it's orchestration, and belongs in `custom/`.

## Worked contrast

Appropriate on the class — a filter that only narrows `self`'s own rows:

```python
@ho_api_filter('published_posts')
def _is_published(self, request):
    user = getattr(request.state, 'user', None)
    visible = self.__class__(published=True)
    if user:
        visible |= self.__class__(author_id=user)
    return self & visible
```

Not appropriate on the class — a hypothetical reporting route that joins
posts, comments and users to compute engagement stats. It needs three
relations, not one, so it belongs in `ho_api/custom/routes.py`:

```python
# ho_api/custom/routes.py
from litestar import get
from blog_demo.blog.post import Post
from blog_demo.blog.comment import Comment

@get('/reports/engagement')
async def engagement_report() -> dict:
    posts = await Post(published=True).ho_acount()
    comments = await Comment().ho_acount()
    return {'posts': posts, 'comments': comments}

routes = [engagement_report]
```
