# Identité fédérée entre plusieurs projets half-orm-gen ("peers")

## Context

half-orm-gen generates one independent CRUD API + admin frontend per
PostgreSQL database. Today, auth is entirely local to a single generated
project: `HO_JWT_SECRET` (a symmetric HMAC secret) is scaffolded once per
project and used both to sign and verify its own JWTs — login/signup
themselves aren't even generated by half-orm-gen, only demonstrated as
example code in `ho_api/custom/routes.py`.

The goal, worked out over an extended design discussion (fully captured in
`planning/identite_federee.md`), is to let several independently deployed
half-orm-gen projects ("peers") share the same user identities, without
introducing any central authority: any peer can issue its own signed
identity tokens, and any other peer can choose to trust it explicitly
(bilateral registration), never by default. Concretely this needs:
asymmetric JWT signing (so verifying a peer's tokens never grants the
ability to forge them), a `peer`/`user`/`login_state` schema, and a
browser-redirect login-delegation flow (OAuth2 Authorization-Code-style)
so a person known to peer A can log into peer B without B ever handling
their credentials.

Key decisions already made in `planning/identite_federee.md` (see that
file for the full reasoning/rejected alternatives):
- No master peer — flat, opt-in, bilateral trust (`peer` table on both sides).
- RS256 keypair generation available at `half_orm gen` time for projects
  that need federation. Small/standalone projects that never federate can
  keep today's simpler HS256 shared-secret scheme — the signing scheme is
  a generation-time choice, not a forced migration; RS256 is only
  necessary once a project actually registers peers.
- Delegation = browser redirect (not server-to-server credential relay),
  with anti-CSRF `state`, redirect_uri validated against the registered
  peer URL, and the returned token's signature checked against the
  specifically-targeted peer's public key (not "any trusted peer").
- New schema `half_orm_meta.identity` (name overridable at generation
  time — a project may already have an incompatible local `user` table).
  Naming it under the `half_orm_meta` prefix means half-orm's own
  `model.classes()` (`.venv/.../half_orm/model.py:720-729`, skips any
  schema starting with `half_orm_meta`) excludes it from generic CRUD
  route generation automatically — no half-orm-gen route-filtering code
  needed.
- Terminology: "peer" (not "silo" — `ResourceSilo`/`SiloRegistry` in the
  frontend already means something unrelated: the per-resource client-side
  data cache).
- The redirect/callback protocol is security-critical (CSRF, redirect
  validation, per-peer signature checks) — generated by half-orm-gen as a
  new module (`ho_api/federation.py`), not left as hand-copied example
  code like today's login/signup.

## Phase 1 — Schema + asymmetric signing

**`half_orm_gen/backend/ho_api/ddl.py`**: add to the DDL, following the
exact existing pattern (`CREATE SCHEMA IF NOT EXISTS`, `CREATE TABLE IF
NOT EXISTS`, `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` for migrations —
see the `field.label_order` example at lines 85-86):

```sql
CREATE SCHEMA IF NOT EXISTS "half_orm_meta.identity";

CREATE TABLE IF NOT EXISTS "half_orm_meta.identity".peer (
    id             uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    name           text NOT NULL UNIQUE,
    url            text NOT NULL,
    jwt_public_key text,
    trusted        boolean NOT NULL DEFAULT true,
    created_at     timestamptz NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE IF NOT EXISTS "half_orm_meta.identity"."user" (
    id             uuid PRIMARY KEY,
    origin_peer_id uuid REFERENCES "half_orm_meta.identity".peer(id),
    name           text,
    email          text,
    password_hash  text,
    first_seen_at  timestamptz NOT NULL DEFAULT CURRENT_TIMESTAMP,
    last_seen_at   timestamptz NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE IF NOT EXISTS "half_orm_meta.identity".login_state (
    state      text PRIMARY KEY,
    peer_id    uuid NOT NULL REFERENCES "half_orm_meta.identity".peer(id),
    return_to  text,
    created_at timestamptz NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

(`login_state` didn't appear in the design doc yet — it's the server-side
store for the anti-CSRF `state` param described in
`identite_federee.md` section 4; consistent with half-orm's "the database
is the source of truth" — not an in-memory dict. Expiry checked at
validation time against `created_at`, e.g. reject if older than 10 min;
opportunistic cleanup of expired rows, no separate cron needed for v1.)

Schema name is a generation-time parameter (default
`half_orm_meta.identity`), same override mechanism as other configurable
generation options — check `half_orm_gen/backend/generate.py` (applies
`HO_API_DDL`) for where to thread this through.

**Asymmetric signing — opt-in alongside the existing HS256 scheme**:
- `half_orm_gen/backend/litestar/v2/scaffold.py:211-216` (`scaffold_api_dir`):
  add a generation-time choice — default unchanged (`secrets.token_hex(32)`
  written to `.env` as `HO_JWT_SECRET`/`HO_JWT_ALGORITHM=HS256`, exactly
  today's behavior for standalone projects); when federation is requested,
  generate an RS256 keypair instead with the `cryptography` package
  (`rsa.generate_private_key` → PEM via `serialization`), write the
  private key to `ho_api/private_key.pem` (gitignored, like `.env` today).
  The public key needs to be easily exportable for OTHER peers to
  register (this project doesn't need its own key in its own `peer`
  table — it signs with the private key directly).
- Add `cryptography` as an explicit dependency in half-orm-gen's own
  `pyproject.toml` (confirmed present only transitively today, not
  guaranteed in a fresh environment — `half_orm_gen/pyproject.toml`).
- `half_orm_gen/backend/litestar/v2/scaffold.py:72-144`
  (`_AUTHORIZATION_TEMPLATE`, generates `ho_api/authorization.py`):
  `_jwt.decode(token, _SECRET, algorithms=[_ALGORITHM])` (line 129) already
  reads `_ALGORITHM` from env — extend it to branch on the configured
  algorithm (`HS256` → shared secret as today, `RS256` → the public key)
  rather than replacing the HS256 path outright.
- `blog_demo` itself doesn't need to federate, so its example signing
  helper (`_sign()` in
  `tests/e2e/scripts/demos/blog_demo/ho_api/custom/routes.py`, generated
  by `demo_blog.sh`'s heredocs — never hand-edited, edit `demo_blog.sh`
  then regenerate via `make demo-blog`) can stay on HS256 unchanged; the
  RS256 signing path only needs an example/doc update once a demo project
  actually exercises federation (Phase 3 verification).
- FastAPI backend (`half_orm_gen/backend/fastapi/v0/`) has no auth
  scaffolding at all today (confirmed) — out of scope here, consistent
  with the FastAPI-parity gap already tracked separately
  (`planning/a_resoudre.md` item 17).

**Pluggable local authentication — DB by default, LDAP or other by peer**

Each peer authenticates its own local users however it chooses (DB
password, LDAP, ...) — this is orthogonal to the cross-peer redirect
protocol (Phase 3), which never needs to know how a peer authenticated
someone locally, only that it comes back with a validly-signed token.
Scope here is the **local** authentication step within a single peer:

- Generate a small pluggable authenticator interface (one function
  signature: given credentials, return a UUID or fail) rather than
  hardcoding a DB check inside `custom/routes.py`'s example login.
- Default generated implementation: DB-based, checking
  `"half_orm_meta.identity"."user".password_hash` (bcrypt/argon2 verify) —
  covers the common case out of the box.
- A per-project config point (env var, e.g. `HO_LOCAL_AUTH=db` vs
  `HO_LOCAL_AUTH=none`) to **disable local DB auth entirely** for peers
  that only want federated sign-in (no local passwords accepted at all —
  in that case `password_hash` stays unused/NULL for every local `user`
  row, and Phase 4's login page shows only "sign in via …" buttons, no
  email/password form).
- LDAP (or any other method) is **not generated** by half-orm-gen itself
  — server URL, bind DN format, attribute mapping are all deployment-
  specific and can't be known generically. Half-orm-gen generates the
  pluggable interface and the DB default; a project needing LDAP
  implements it in `custom/` against that same interface and points
  `HO_LOCAL_AUTH` at it. Consistent with the existing convention that
  business-specific auth logic is custom code, not generated.

## Phase 2 — Peer management (admin backend + Angular UI)

Follow the exact `ho_admin.py` pattern (`half_orm_gen/backend/litestar/v2/ho_admin.py`):
a small `HoIdentityModels`-style wrapper (mirroring `HoApiModels` in
`models.py`) over `"half_orm_meta.identity".peer`, new handlers in the
same style as `make_ho_admin_handlers` (`_check_admin` guard, halfORM
`ho_aselect`/`ho_ainsert`/`ho_aupdate`/`ho_adelete`, no raw SQL) for:
`GET/POST/PUT/DELETE {prefix}/ho_admin/peer` (register/list/update
trust flag/remove a peer — name, url, public key). Also a small public,
unauthenticated `GET {prefix}/auth/peers` (name + url only, `trusted =
true`) for the login page to list "sign in via …" options.

Angular only (Svelte has no admin UI at all, confirmed) — new panel in
`half_orm_gen/frontend/angular/v19/_ho_admin.py`, following its existing
per-section pattern, to add/list/revoke peers.

## Phase 3 — Federation login flow (generated `ho_api/federation.py`)

New generated module (same scaffolding mechanism as `authorization.py`),
implementing the protocol from `identite_federee.md` section 4:
- `GET /auth/login?peer=<name>` — generate `state`, insert into
  `login_state` (peer_id, return_to), redirect to
  `{peer.url}/auth/login?redirect_uri={this_peer_url}/auth/callback&state={state}`.
- `GET /auth/callback?token=<jwt>&state=<state>` — look up `state` in
  `login_state` (exists, not expired, single-use — delete on read),
  verify `token`'s signature against **that specific peer's**
  `jwt_public_key` (from the `peer` row the state points to), upsert
  `"half_orm_meta.identity"."user"` (`origin_peer_id` = that peer,
  `last_seen_at` refreshed), mint a local session JWT (this project's own
  private key), redirect to `return_to` with the local token.
- Local peer's own `/auth/login` (whatever the developer wrote in
  `custom/routes.py`) needs no change — it's the target of `redirect_uri`
  when THIS peer acts as the identity source for someone else.

## Phase 4 — Frontend: peer list on login + callback route

Both frameworks currently embed the login form directly in the shared
shell/layout component (Angular: `_app_component` in `_app_shell.py`
~lines 238-371; Svelte: `_layout` in `svelte.py` ~lines 591-851) — no
separate login page with real logic exists yet (`_login_component`/
`_login_page` are placeholders). Add:
- Fetch `{API_BASE}/auth/peers` and render "Sign in via …" buttons
  alongside the existing email/password form, each linking to
  `{API_BASE}/auth/login?peer=<name>`. The email/password form itself is
  only shown if this peer has local DB auth enabled (`HO_LOCAL_AUTH=db`,
  Phase 1) — exposed via `/ho_meta` or a small flag on `/auth/peers`'
  response, so a federation-only peer's login page shows just the peer
  buttons, no local form.
- New callback route: Angular — append to `_app_routes`
  (`_app_shell.py:807-845`) a lazy route `auth/callback` → new
  `AuthCallbackComponent` (new file/function, reads the token query param,
  calls a new `AuthService.setToken()`-equivalent, redirects to the
  original destination). Svelte — new file-based route written via
  `self._write(routes_dir / 'auth' / 'callback' / '+page.svelte', ...)`
  near the existing `_write` calls around `svelte.py:2261-2263`.
- Token storage/attachment is already framework-agnostic (`sessionStorage`
  + manual `Authorization: Bearer` header in both `AuthService`/`AuthState`
  — Angular `_app_shell.py:79-91`/`594`/`691`, Svelte `svelte.py:215-226`/
  `662`/`950`) — the callback page just needs to store the token the same
  way `loginWithEmail`/`signupUser` already do, no new mechanism needed.

   `blog_demo` still scaffolds and logs in exactly as today (HS256
   unchanged, default). Separately, generate a fresh throwaway project

   all (login delegation is meaningless with a single project) — and it
   needs to be reproducible via `make`, not a one-off manual copy, to
   match how every other demo-based check in this project works
   (`make demo-blog`/`demo-blog-clean`/`demo-blog-run`). Add this as part
   of the implementation itself (Phase 2/3), not just a verification
   afterthought: parameterize `tests/e2e/scripts/demo_blog.sh` (project
   name/port/DB name currently hardcoded — check exact spots) so it can
   scaffold a second instance (e.g. `blog_demo_peer` on a different port,
   reusing the same schema — federation doesn't care about the business
   domain, only the identity plumbing), plus matching Makefile targets
   (`demo-blog-peer`, `demo-blog-peer-clean`, `demo-blog-peer-run`,
   mirroring the existing `demo-blog-*` targets). Then: register each
   instance as the other's peer (bilateral), attempt login on B for a
   person who only has an account on A, confirm the browser redirect
   round-trip and that B's `"half_orm_meta.identity"."user"` row is
   created with the correct `origin_peer_id`.
3. Negative tests: expired/reused `state` is rejected; a token signed by
   an untrusted/unregistered peer is rejected; a `redirect_uri` not
   matching the registered peer URL is rejected.
